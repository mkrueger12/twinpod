Topic Context

Twinpod is a planned orchestration layer on top of OpenCode (the open-source, model-agnostic terminal coding agent). Its core loop: watch Linear for new issues, run a defined workflow against each with OpenCode, and produce a reviewable PR — the same shape as OpenAI's Symphony concept. The repo is empty/greenfield; this document grounds in the user's description plus external research, not a codebase.

The defining strategic fact: the concept is already validated and commoditized.

  • OpenAI Symphony (open-sourced ~Apr–May 2026) is near-identical: an open SPEC.md + Elixir reference impl that polls a tracker, namespaces an isolated workspace per issue, dispatches an agent read→plan→code→PR, with lifecycle hooks making branch/PR creation idempotent. Reported ~500% increase in landed PRs at some OpenAI teams. The spec is public — anyone can build this.
  • Tembo is a live competitor that already orchestrates OpenCode (among others) from Linear, with a self-host option. Charlie Labs ships "Daemons" — always-on agents that keep PRs/CI/issues moving after creation. Devin, Cursor, Jules, Copilot coding agent, Codex cloud, Claude Code Remote Tasks, Factory all do some flavor of issue/label/mention → PR.
  • The hard parts everyone agrees on: the Linear plumbing and headless OpenCode invocation (opencode serve + @opencode-ai/sdk) are solved. What remains hard — and unowned — is quality gating, sandbox/concurrency isolation, cost control, and keeping human review cost low.

The market's actual pain (consensus across competitor postmortems and practitioner complaints): autonomy claims outrun reliability — only ~28% of agent PRs merge instantly, ~72% need rework; agents are "optimistic and produce something even when they shouldn't"; broken CI loops let bad work through until a human cleans up; long-running loops are expensive; and review burden — not generation — is the real cost. The winning posture is human-in-the-loop plus strong CI gating.

Every survivor below is chosen to land on execution differentiation — because the concept itself is no longer a moat. The seven compose into a single opinionated pipeline:

Topic Axes

  • Work intake — how issues enter: polling vs webhooks, filtering, triage, dedup/idempotency, scoping out low-value work.
  • Workflow execution — what the agent does per issue: plan → code → test → PR, workflow definition, agent roles & tool permissions, TDD/verification loops.
  • Orchestration & concurrency — running a fleet: fan-out, sandbox/isolation, supervision, concurrency caps, durable execution, exactly-once.
  • Human-in-the-loop & control — approval/plan gates, review queues, escalation, observability, cost budgets per run.
  • Output & feedback loop — PRs, branch/worktree management, results back to Linear, CI integration, learning from merged/rejected outcomes.

1

CI-Green-Gate — humans only ever see green PRs

Top pick
Axis · Output & feedback loop Confidence 88% Complexity · Medium
Description

Twinpod never opens or promotes a PR until the full test/lint/typecheck/build suite passes inside the ephemeral sandbox where the run executes. The inner loop is: implement → run the suite in-sandbox → on failure, feed the structured failure back to the implementer agent → repeat up to a bounded number of cycles with a hard cost/time cap → only on green push the branch and open a draft PR. If green isn't reached within budget, post the last failure to the Linear issue and close the run instead of shipping broken work. PR merge rate — not PR open rate — becomes the product's north-star metric.

Basis
external: Market signals — "broken CI loops let bad work through until a human cleans up," "~28% of agent PRs merge instantly / ~72% need rework," and "opinionated per-issue workflow with built-in CI/test gates before PR" named as an explicit unmet gap; the CI/CD analogy "loop until green = CI gate run inside the agent before PR"; E2B/Daytona sandboxes available as the execution substrate.
Rationale

Every named competitor — including Symphony — opens the PR and relies on CI after the fact. Moving the gate inside the run means every PR a human ever sees was green at creation, which attacks the 72% rework rate and the review-burden cost simultaneously. It is the clearest single-sentence value proposition in the whole candidate set: "humans only review green PRs."

Downsides
Requires a reliable sandbox and a repo whose test suite runs headlessly — real setup cost, plus flaky-test handling. The budget cap means some issues end with no PR (a "stuck" outcome that still needs a human). And it only catches mechanical failure (CI red), not semantic or design wrongness — green ≠ correct.

Opinionated RED-First TDD Spine

Axis · Workflow execution Confidence 82% Complexity · Medium
Description

Make TDD the one opinionated workflow rather than a configurable option: no implementation code until a @test agent has written a failing test and CI confirms it fails for the right reason. A failure taxonomy gates progress — MISSING_BEHAVIOR / ASSERTION_MISMATCH are valid RED states (proceed to implement to GREEN); TEST_BROKEN / ENV_BROKEN are hard-abort conditions that pull an "andon cord" — halt the run and escalate to Linear with a diagnostic instead of thrashing. If no meaningful test can be written because the issue is too vague, the agent posts a clarifying question rather than guessing.

Basis
external: The ppries OpenCode+Linear gist (the closest existing prototype on this exact stack) implements precisely this RED/GREEN pipeline and failure classification; Toyota's jidoka/andon principle (stop the line on a defect rather than passing it downstream) supplies the escalation action; grounding names "opinionated per-issue workflow" as the unmet gap.
Rationale

A behavioral contract before code eliminates whole categories of plausible-but-wrong output before a token is spent implementing. Being deliberately opinionated is the counter-position to Tembo's agent-agnostic breadth — the grounding says twinpod's edge must be "opinionated/single-stack simplicity," and a mandatory test-first spine is that edge made concrete.

Downsides
TDD doesn't fit all work — config changes, docs, exploratory spikes, and pure UI tweaks need an escape hatch or per-issue-type workflow selection. Writing a good failing test for a vague issue is itself hard, and strong opinionation narrows the set of issues twinpod can take.

The Outcome-Learning Flywheel

The moat
Axis · Output & feedback loop Confidence 65% Complexity · High
Description

Treat every merged, rejected, or abandoned PR as a labeled datapoint (issue type × area × workflow × model × outcome + review comments) stored locally and owned by the team. That store drives three compounding behaviors: (a) extract why-rejected lessons and inject the relevant ones into future runs touching the same files/labels; (b) route each issue to the cheapest capable model based on historical confidence; (c) accrue a per-surface trust score so well-performing areas graduate from plan-gated to unattended, while shaky areas stay gated. Run 1 is noisy and expensive; run 500 is calibrated and cheap.

Basis
reasoned: The grounding names "learning from merged/rejected outcomes" as an explicit unmet capability, and the one asset closed competitors (Devin/Charlie/Tembo) structurally cannot expose to self-hosters — they are stateless across runs and won't surface a team's private outcome data. The adaptive-immune-memory analogy holds (a recorded signature pre-shapes a faster future response), and Devin's reported 34%→67% merge jump shows feedback pressure works. Reframing the trio of routing/trust/triage as one store resolves the redundancy a critique pass flagged when they were proposed separately.
Rationale

Because the concept is commoditized (Symphony's spec is public; Tembo already runs OpenCode), feature parity is not a moat. A compounding local outcome store is — it makes the longest-running deployments the cheapest and most autonomous, and that accumulated history doesn't transfer if a team switches tools. This is the strategic answer to "why twinpod instead of forking Symphony."

Downsides
Sequencing is the catch: it needs the core loop running and producing labeled outcomes first, so it delivers near-zero value at launch — a fresh-context critique pass rated it "weak" on exactly this point. It also risks overfitting to noisy labels, demands conservative trust-graduation (higher autonomy isn't always better), and is the highest build complexity in the set. Treat it as the v2 bet you architect toward from day one, not the thing you ship first.

Plan-Approval Gate in Linear

Axis · Human-in-the-loop & control Confidence 84% Complexity · Small
Description

Before writing any code, the agent posts a structured implementation plan (files to touch, approach, test strategy, risk flags) as a Linear AgentSession comment and moves the issue to a "Plan Review" state. Execution is gated on a human approving — an emoji reaction, a label, or a status move. The eventual PR is then reviewed against a contract the human already signed off on. No new UI: Linear is the approval surface. A configurable timeout expires the session gracefully if no one responds.

Basis
external: Modeled on "registered reports" in science — reviewers approve the methodology before the expensive experiment runs. Grounding consensus: "review burden is the real cost, not generation," "HITL + strong CI gating is the winning posture," and Swarmia's "five levels of autonomy — higher isn't always better." The ppries gist already includes a plan phase, confirming workflow fit.
Rationale

Reviewing a 200-line plan costs an order of magnitude less than reverse-engineering a 2,000-line diff, and it kills wrong-direction work before it exists — the single most direct lever on the stated #1 cost. It's also the trust on-ramp: teams not ready for full autonomy can adopt twinpod at the plan-gate level and graduate later (this is what idea #3's earned-trust score eventually automates).

Downsides
Adds a human-latency step that defeats "set and forget" for trivial issues — it should be skippable per issue-type, or once trust is earned. A rubber-stamped plan gives false confidence, and some plans are wrong in ways only visible once code exists.

Issue Fitness Triage

Axis · Work intake Confidence 80% Complexity · Small
Description

A cheap-model (Haiku-class) pre-flight classifier scores every incoming issue on clarity of acceptance criteria, scope (single vs multi-concern), testability, blast radius, and dedup against open work. Below threshold → an automated "needs clarification" comment plus a re-label, not a dispatch. Above threshold → the issue enters the run queue, optionally tagged with which workflow fits. Refusal is the feature.

Basis
external: Grounding names "scoping/triage to avoid low-value PRs" as an unmet need and "agents are optimistic and produce something even when they shouldn't" as the core failure; the ppries gist already uses a Haiku @pm reader; the ER acuity-triage analogy (Emergency Severity Index) maps cleanly onto pre-dispatch scoring.
Rationale

Token cost and abandoned-PR fatigue are the top reasons teams quit coding agents. A strict intake gate protects both the budget and the team's trust by never running on work no agent should touch — the cheapest possible quality lever, applied before any tokens are spent on implementation.

Downsides
False negatives (a deferred issue that was actually fine) erode trust in the other direction; thresholds need tuning — which is exactly the signal idea #3's outcome store eventually supplies. Adds one LLM call to every issue (cheap, but non-zero).

Issue-ID = Durable Workflow ID

Axis · Orchestration & concurrency Confidence 80% Complexity · Medium
Description

Make the dispatch unit a durable workflow keyed to the Linear issue ID, so the orchestrator gets exactly-once dispatch and crash-safe resume essentially for free: a duplicate webhook or poll hits the same workflow ID and no-ops; a crash mid-run re-attaches to the existing workflow instead of double-creating branches/PRs; and hours-long approval gates (idea #4) survive container restarts because the wait is durable state, not memory. The genuine architecture decision to debate: lean on Linear/tracker-state as the durable log (delete the job-queue tier entirely, Symphony-style namespacing) versus adopt Temporal (TS SDK, built-in retries/timeouts/idempotency).

Basis
external: Symphony's namespace-based dedup ("each issue → isolated workspace dir named by issue ID; tracker state is the persistence layer enabling resume/restart"); Temporal's explicit guarantees ("idempotency so retries don't double-create PRs; approvals reattach after crashes"); the grounding flags double-PR/idempotency as a specific named hazard.
Rationale

Get this primitive right once and concurrency, crash recovery, and the double-PR hazard are solved for every feature layered on top — approval gates, cost checkpoints, multi-agent fan-out all inherit exactly-once and resumability. The Linear-vs-Temporal choice shapes the entire architecture, which is precisely why it's worth a deliberate early decision rather than a default.

Downsides
Temporal adds an operational dependency a solo founder may not want to run. Pure Linear-as-state is simpler to stand up but pushes more idempotency burden into your own code and is bounded by Linear's API rate limits and the shape of its state model.

Event-Driven Intake — the Webhook-Ack Firewall

Axis · Work intake Confidence 78% Complexity · Small

Challenges your stated approach

You described twinpod as polling Linear for new issues. The research points to Linear's first-class AgentSession webhooks as the better-supported pattern (it's what Symphony, Tembo, and Cursor use). This idea is included specifically because it pushes back on the polling assumption — but polling is a legitimate simpler v1, so the tradeoff is laid out rather than decided.

Description

Replace naive polling with Linear's AgentSession webhooks, and architect the handler as a pure enqueuer: validate the payload, enqueue a durable job keyed to the issue ID (idea #6), and post a "received" activity — all in well under 200ms, comfortably inside Linear's 5-second-ack / 10-second-activity window. The actual coding run happens in an async worker that can take hours. This same ack-then-enqueue contract becomes the single intake boundary for every future trigger source — a GitHub mention, a Slack signal, a Sentry alert all enter the same way.

Basis
direct: The user's stated v1 is "polling Linear for new issues." external: The grounding documents Linear's AgentSession webhook model and the hard 5s/10s ack constraint (miss it and the session is silently marked unresponsive), and recommends event-driven intake as the supported pattern competitors use; the Leverage frame adds the "one intake contract for all sources" payoff.
Rationale

Missing the 10-second window silently degrades every issue and makes twinpod look broken — so the intake contract is a correctness gate, not just an optimization. Getting it right once turns every future trigger source into a single-file integration instead of a new reliability surface.

Downsides
Webhooks require a public endpoint plus retry/dedup handling that polling avoids — and Symphony deliberately chose polling to skip webhook infra, so polling is a defensible simpler starting point. This is a real tradeoff, not a slam dunk; it pairs naturally with idea #6's durable-job idempotency to handle Linear's webhook retries.
